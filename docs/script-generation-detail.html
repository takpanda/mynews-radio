台本生成の全体像

generate_script.py は、記事要約 DB から取得したニュースをもとに、2フェーズで台本 JSON（script.json）を生成します。その後 review_script.py が5ディレクターによるLLMレビューと改訂を行います。

エントリポイント

def generate_script(output_path: str, program_name: str = "ニュースのとなり", news_source: str | None = None) -> int
引数型説明
output_pathstrscript.json の書き出しパス（例: data/episodes/42/script.json）
program_namestr番組名。既定値 "ニュースのとなり"。news_source="hatena_bookmark" 時は "テックニュース" を渡す
news_sourcestr | None記事絞り込み用ソース指定。None で全ソース対象

戻り値: 生成された台本行数（int）。0 の場合は失敗としてパイプラインを中断。

使用する Ollama 呼び出し回数

フェーズ呼び出し数備考
Narrative Arc 生成1回失敗時はスキップし Arc なしで続行
台本生成（Writer）1〜3回Auto-Lint 合格まで再生成。SCRIPT_LINT_RETRIES 環境変数で最大試行数を制御（既定3）
5ディレクターレビュー（review_script.py）5 + 1 = 6回ディレクター4名 + radio + synthesis の計6回

1エピソード生成あたり、最小2回・最大9回の Ollama 呼び出しが発生します。モデルのレスポンスタイムによりますが、標準的な構成では合計5〜15分程度を要します。

78.0

Step 1: Narrative Arc 生成（Architect）

台本を書く前に、まずプロデューサー役の LLM がエピソード全体のストーリー構造（Narrative Arc）を設計します。これにより、複数記事をバラバラに並べるのではなく、共通テーマで貫かれた一本の物語として番組を構成できます。

処理フロー

  1. DB から取得した要約一覧（JSON）を generate_narrative_arc.md プロンプトへ埋め込む
  2. Ollama を呼び出し、Arc JSON を取得する
  3. Arc の article_order に従って要約一覧を並び替え（_reorder_summaries()）
  4. Arc 情報をプロンプトへ注入するセクション文字列（_build_narrative_arc_section()）を生成し、台本生成ステップへ渡す

Arc 生成が失敗した場合（Ollama がタイムアウト、または article_order が無効な場合）は、Arc なしで続行します。この場合、記事は元の重要度順で台本に渡されます。

Arc JSON の出力フォーマット

{
  "theme": "エピソードを貫く共通テーマ（20〜40文字）",
  "theme_description": "テーマの補足説明（60〜100文字）。台本ライターへの具体的な指示",
  "article_order": [3, 1, 5, 2, 4],          // 記事IDの並び替え後の順序
  "bridges": [
    {
      "from_article_id": 3,
      "to_article_id": 1,
      "bridge_text": "この2記事を繋ぐ文脈・対比・共通点（30〜60文字）"
    }
  ],
  "discussion_article_id": 1,               // discussion で深掘りする記事ID
  "discussion_reason": "選んだ理由（40〜80文字）"
}

記事順序設計の原則

  • 起承転結 を意識し、中程度のインパクトのニュースで掴み始める
  • 最重要トピックは冒頭に使い切らず、中盤以降に配置する
  • discussion_article_id の記事は最後から2番目の news として配置し、discussion セクションへ自然に繋げる
  • 複数記事に共通する「緊張・対比・連鎖」（例: "便利さ vs. リスク"）をテーマとして発見する

Bridge（橋渡し）の活用

Arc の bridges は、記事間を繋ぐ文脈情報です。_ensure_transitions() がこの情報を参照し、通常の transition フレーズ（「続いては〜」）の代わりに Contextual Bridge transition（「{bridge_text}。さて、{topic}の話題です。」）を生成します。Arc なし・Bridge なしの場合は通常フレーズに fallback します。

Step 2: 台本執筆（Writer）

Narrative Arc の情報を注入したプロンプトを使い、Ollama に台本 JSON を生成させます。Auto-Lint が合格するまで最大3回再生成します。

台本生成のメインループ

_MAX_LINT_RETRIES = int(os.getenv("SCRIPT_LINT_RETRIES", "3"))
current_prompt = base_prompt                 # Arc 注入済みプロンプト

for lint_attempt in range(1, _MAX_LINT_RETRIES + 1):
    response = client.generate_json(current_prompt)
    if response is None or not list型:
        break                                # 致命的失敗

    lint_errors = lint_script(response["lines"], ...)
    if not lint_errors:
        break                                # ✅ 合格

    if lint_attempt < _MAX_LINT_RETRIES:
        current_prompt = _build_correction_prompt(base_prompt, response["lines"], lint_errors)
    # 最終試行で失敗してもそのまま使用（最善の結果を保持）

生成後の後処理

  1. フィールド正規化: speaker を "male"/"female" に強制、section を有効値に強制、text の前後空白を除去
  2. プレースホルダー除去: 〔...〕形式（全角ブラケット）を正規表現で削除
  3. transition 補完: _ensure_transitions() を実行（後述）
  4. ファイル書き出し: Path.write_text() で UTF-8 JSON を書き出し
  5. 記事 "used" マーク: 採用した記事を article_service.mark_articles_used() で次回から除外

Ollama クライアント設定

設定項目値説明
エンドポイントPOST /api/generateOllama の生成 API
format"json"Ollama に JSON モードを指示（structured output）
streamfalseストリーミング無効（完全レスポンス待ち）
HTTP タイムアウト120 秒大規模な台本生成に対応した長めのタイムアウト

JSON パース直接 → 抽出 の2段階まず json.loads()、失敗したら {...} 部分抽出を試みる

reasoning モデル（thinking フィールドを持つモデル）に対しては、data["response"] が空の場合に data["thinking"] へ fallback します。

82.5

自動トランジション挿入（_ensure_transitions）

_ensure_transitions(lines, summaries, arc) は、LLM が生成した lines を後処理し、記事 ID の切り替わり境界に必ず transition 行が存在することを保証します。

処理ロジック

話者選択ロジック（_pick_speaker）

トランジション行の話者は _pick_speaker(result, section) が以下のルールで決定します：

  1. 直前の news/discussion 行（最大3件）の話者パターンを取得
  2. 同一話者が2連続以上 → 強制的に相手話者
  3. 直近3行でバランスが2:1以上に偏っている → 少数側の話者
  4. それ以外 → 直前コンテンツ行と交互（alternate）

フレーズ重複回避（_pick_phrase）

30種のフレーズから乱択しますが、直前に使ったフレーズと同一のものを避けるため、used_indices 辞書で最後に選んだインデックスを管理します。フレーズが1種しかない場合は重複回避を無効化します。

Auto-Lint & 再生成ループ

lint_script(lines, program_name, bridges) は生成された台本に対して品質チェックを実行し、問題点のリストを返します。空リストを返せば合格です。合格するまで最大3回再生成します（SCRIPT_LINT_RETRIES）。

Lint ルール一覧

ルールID種別チェック内容合格条件
INTRO_FORMAT ERROR intro 最初の行が番組名で始まるか 「{番組名}」の時間です で始まること
INTRO_FORMAT ERROR intro セクション自体が存在するか intro 行が1行以上あること
INTRO_LINEUP WARN intro にラインアップを示す表現があるか 「ラインナップ」「本日」「今日」などキーワードが2個以上
OUTRO_LENGTH ERROR outro が十分な行数か outro が2行以上あること
TRANS_VARIATION ERROR transition 行の文字列が類似しすぎていないか 任意の2行間の Levenshtein 類似度が 0.7 未満（LINT_TRANS_SIMILARITY で変更可）
TRANS_CONTEXT ERROR Narrative Arc Bridge 存在時に汎用的すぎる transition を使っていないか 「次の話題」「続いては」「次は」「では次」で始まる transition が2個目以降に出現しないこと
— ERROR discussion が全 news の後に配置されているか discussion 最初の行インデックス > news 最後の行インデックス
— ERROR テキストの重複 同一テキストが2行以上現れないこと
— ERROR 禁止フレーズの存在 下表「禁止フレーズ」が1件も検出されないこと
— ERROR 数字なし「数字で見ると」系フレーズ 「数字で見ると」使用時、同行に数字・数量表現があること
TRUNCATED_TRANS WARN transition 行の不完全テキスト 「……」連続2個以上の末尾 / 5文字未満の transition を警告
— ERROR 記事 ID 参照の混入 「記事XX」「（ID: XX）」などの記事ID参照表現がないこと
— ERROR プレースホルダー〔...〕の残存 全角ブラケット〔...〕がテキストに残っていないこと

禁止フレーズ（Forbidden Phrases）

以下のフレーズがテキスト中に含まれていると Lint エラーになります（プロンプト内の Forbidden Phrases テーブルと同期）：

禁止フレーズ理由
ここで大事なのは、背景ですよね / ここで大事なのは背景ですよね抽象的な逃げ。具体的な背景事実を述べるべき
といったところです。 / といった内容です。フィラー。台詞の締まりがなくなる
一見シンプルに見えますが、実は構造的な問題があります田村の口癖。使い回しで不自然になる
これは感情論だけでは片づけられません田村の口癖。同上

「口癖」として登録されているフレーズは Forbidden Phrases（完全禁止）と二重管理されています。これは、禁止フレーズとして登録されても LLM が出力してしまうケースに対応するためです。

修正プロンプト（_build_correction_prompt）

Lint 不合格時は、元のベースプロンプトに以下を追記した修正指示付きプロンプトを生成して再生成します：

# ⚠️ 前回の生成で以下の品質問題が検出されました。これらをすべて修正して再生成してください。

- [INTRO_FORMAT] introの1行目が「ニュースのとなり」の時間です」で始まっていません
- ...（Lint エラーリスト）

## 前回生成した lines（修正対象）:
```json
[ ... ]
```

上記の問題を修正した完全な台本 JSON を出力してください。

MC キャラクター設定

番組は2名のMCによる掛け合い形式で進行します。それぞれ明確な役割分担があり、プロンプトで厳密に定義されています。

SPEAKER KEY: "male"

田村 誠（たむら まこと）

38歳 / 元新聞記者・ニュースキャスター

  • 担当: 事実・数字・データの正確な伝達
  • 担当: 技術的な仕組みや構造の説明（平易な言葉で）
  • 担当: 制度・背景・歴史的経緯の解説
  • 担当: 論点の整理と問題の構造化
  • 禁止: 感情・共感・リスナー視点の台詞
  • 性格: 落ち着いていて理性的。難しい話を噛み砕く
  • 話し方: 短く、歯切れよく。やわらかい言葉で要点
SPEAKER KEY: "female"

山口 麻衣（やまぐち まい）

31歳 / ラジオパーソナリティ・生活情報コメンテーター

  • 担当: 生活者・消費者の目線で影響を具体化する
  • 担当: リスクや不安を代弁し、リスナーの疑問を言語化
  • 担当: 田村の説明を「つまりこういうことですよね？」と噛み砕く
  • 担当: 生活者視点からの分析・異論・別角度の提起
  • 禁止: 技術構造・制度・データの詳細解説
  • 性格: 明るく率直。共感力が高く、疑問をそのまま言葉にする
  • 話し方: やわらかく親しみやすい。少しテンポが速い

役割分担の鉄則

  • 田村が説明した内容を山口が単に別の言い方で繰り返す行は禁止。必ず視点・情報が追加されていること
  • 山口の異論は「技術的事実への反論」ではなく「生活者視点からの別角度の提示」であること
  • 山口からの問題提起に対して田村が事実・構造で受け止める相互作用を許容する
  • 田村は「感情論での受答」は禁止。山口は「技術構造の詳細解説」は禁止

セクション構成ルール

台本の各行は section フィールドによって5種類のセクションに分類されます。セクションの並び順と行数に関する規則があります。

セクション一覧

sectionarticle_id行数の目安役割・制約
intro null 2〜3行 1行目は必ず 「{番組名}」の時間です。{独自の一言} で始める。ラインアップ紹介を含むこと
transition 次の記事の article_id 1行 話題の切り替えブリッジ。_ensure_transitions() により記事 ID 境界に自動保証。news 行のテキスト内に埋め込まない
news 対象記事の article_id 記事あたり2〜6行 記事の事実概要 → MC 掛け合い → 生活影響・リスク解説。難易度3（技術/セキュリティ/金融）は田村が仕組みを解説し、山口が影響を語る追加ループあり
discussion 最高重要度記事の article_id 4〜8行 全 news 紹介後に最高重要度記事を深掘り。discussion_article_id（Narrative Arc 指定）の記事を使用。全 news より後に配置すること（Auto-Lint でチェック）
outro null 2行以上 当日の話題まとめ・締め。最低2行（Auto-Lint: OUTRO_LENGTH）

セクション配置順序

[intro] × 2〜3
  ↓ transition（記事1へ）
[news 記事1] × 2〜6
  ↓ transition（記事2へ）
[news 記事2] × 2〜6
  ↓ ...（全記事）
  ↓ transition（discussion 記事へ）
[discussion] × 4〜8
  ↓
[outro] × 2以上

delivery フィールド

各行には任意の delivery フィールドが付与されます。音声合成のパラメーター調整に将来利用できます（現在は記録のみ）。

値場面
neutral通常（既定値）
emphasis強調・重要事実
thoughtful思慮深い・discussion
questioning疑問・問いかけ
warm親しみ・outro

5ディレクターレビュー（review_script.py）

review_script(source_script_path, output_dir) は、生成済み台本に対して5種類のディレクターキャラクターによるLLMレビューを実施し、その結果を統合した改訂版 script.json を生成します。この処理は非致命的（失敗してもメインエピソードには影響なし）です。

ディレクター 5 名

天才ディレクター (genius)

番組全体の構成・テーマの深さ・MC 間の化学反応を高い視座で評価する

初心者ディレクター (beginner)

ラジオ初心者目線。分かりにくい表現・専門用語・説明不足を指摘する

心配性ディレクター (worried)

リスク管理視点。放送禁止表現・誤情報・炎上リスクを保守的に評価する

ポジティブディレクター (positive)

良い点を積極的に評価し、改善点より強みを伸ばす提案を行う

ラジオ専門ディレクター (radio)

ラジオメディアの特性（耳だけ・間・テンポ・口語）に特化した評価を行う

処理フロー（2ステップ）

ディレクターレビュー JSON フォーマット（例）

{
  "overall_score": 7,               // 1〜10点
  "issues": [
    {
      "line_index": 12,
      "severity": "high",           // high / medium / low
      "description": "田村が共感的な台詞を言っており役割から逸脱している",
      "suggestion": "「一般的には〜という構造があります」などに修正"
    }
  ],
  "strengths": ["intro の構成が自然", "transition のバリエーションが豊富"],
  "summary": "全体的に役割分担が明確で質が高い。いくつか口語表現の硬さが残る"
}

戻り値

キー型説明
revisedbool改訂版 script.json が書き出されたか
review_countint成功したディレクターレビューの数（0〜5）
revision_summarystr改訂内容の要約テキスト（LLM 生成）
lines_countint改訂版 lines の行数（0 なら未生成）

synthesis（統合）が有効な改訂版を返さなかった場合（lines が空、または None）は revised=False で返り、メインエピソードの script.json は上書きされません。
