システム概要

mynews-radio は、ニュース記事の収集から音声番組の配信までを完全ローカル環境で完結させる「パーソナルニュースラジオ生成システム」です。クラウドのLLM APIや商用TTSを一切使わず、ローカルネットワーク上の Ollama（LLM推論サーバー）と VOICEVOX / AivisSpeech（音声合成エンジン）を呼び出して番組を生成します。

主要な機能

  • 記事取得 — はてなブックマーク系API（news.beeworks.cc）、はてな ホットエントリRSS、Yahoo!ニュースRSS の3ソースから記事を収集し、SQLite に upsert
  • 要約生成 — Ollama で各記事を日本語要約し、カテゴリ・重要度（1〜5）・難易度（1〜3）を付与
  • 台本生成 — Narrative Arc（番組全体の構成案）を先に生成した上で、男女2人のトーク形式のラジオ台本（JSON）を生成。トランジション自動挿入と自動Linterによる品質チェック・再生成を実施
  • 4ディレクターレビュー — 「天才」「初心者」「心配性」「ポジティブ」の4キャラクターによるLLMレビューを集約し、改訂版台本から別エピソードを並行生成（非致命的処理）
  • 音声合成 — VOICEVOX互換API（audio_query → synthesis の2段階）でセリフごとにWAVを生成。読み置換テーブル適用、トランジションジングル/無音挿入、サンプリングレート正規化を実施
  • エピソード組み立て — ffmpeg でWAVを結合し、オープニング/エンディングジングル（フェード付き）を付加して episode.mp3 と metadata.json を出力。各セリフに再生開始秒（start_time）を付与
  • 配信 — FastAPI が HTTP Range 対応で音声をストリーミング配信し、Next.js 製のWebフロントで再生・台本閲覧・手動生成が可能

全体アーキテクチャ

システムは Docker Compose 上の2コンテナ（api / web）と、ローカルネットワーク上の外部サービス（Ollama・TTSエンジン）、およびインターネット上のニュースソースで構成されます。永続データはホストの ./data ディレクトリ（SQLite DB・エピソード成果物・ジングル・ログ）にボリュームマウントされます。

データモデル（ER図）

永続化は単一の SQLite ファイル（data/mynews-radio.db）で行います。スキーマは app/db/schema.sql に定義され、FastAPI 起動時に冪等適用（CREATE TABLE IF NOT EXISTS）された後、ALTER TABLE による追加マイグレーション（articles.difficulty、episodes.phase、generating 部分一意インデックス）が安全に実行されます。

ステータス遷移

テーブル列取りうる値意味
articlesstatusnew → summarized → used / error取得直後 → 要約済み → 台本に採用済み。本文不足やJSON解析失敗時は error
episodesstatuspending / generating → completed（または done）/ failedrun_daily・API は completed、orchestrate.py は done を完了値として使用
episodesphasestart → import → summarize → generate_script → review → review_done → synthesize → build → completeAPI 経由生成時の進捗表示用。フロントがポーリングして表示

SQLite の database is locked 対策として、EpisodeService には指数バックオフ付きリトライデコレーター（最大5回）が実装されている。また「同一日付で generating 状態のエピソードは1件のみ」という部分一意インデックスにより、手動生成の二重実行が DB レベルで防止される。

手動生成フロー（Web → API）

Web フロントの「生成」ボタンから始まる手動生成は、Next.js API Route のプロキシを経由して POST /generate を呼び出します。API はエピソードレコードを作成して即座に応答し、実際のパイプラインは asyncio スレッド（uvicorn 配下）またはデーモンスレッドで非同期実行されます。フロントは GET /episodes/{id} をポーリングし、phase 列の更新で進捗を表示します。

モジュール詳細

services/ — 外部サービスクライアントとドメインサービス

モジュール責務
ollama_client.pyOllama /api/generate を format: json・タイムアウト600秒で呼び出す。reasoning 系モデルが本文を thinking フィールドへ出力するケース（<|channel|>output タグ）からの抽出、空レスポンス時の「JSONのみで回答せよ」プロンプトによる自動リトライ（最大2回）、コードフェンス除去付き JSON パースを実装
voicevox_client.pyVOICEVOX 互換 API クライアント。POST /audio_query（enable_katakana_english=1）→ POST /synthesis（enable_interrogative_upspeak=1）の2段階でセリフ1行をWAV化。speaker 名（male/female）→ 話者IDのマッピング、/version によるヘルスチェックを提供。AivisSpeech も同一クライアントで扱う
hatena_fetcher.py3種のニュースソース取得。① news.beeworks.cc API（Markdown 形式 content を正規表現でタイトル・元記事URL・ブックマーク数・ユーザースコアにパース。データがある直近日まで最大数日分さかのぼり取得）② はてな hotentry/all RSS ③ Yahoo!ニュース RSS。RSS 系は trafilatura で本文を抽出
article_service.py記事の upsert（URL一意で重複判定）、status=new の取得、要約結果の保存、台本用要約の取得（importance_score >= MIN_IMPORTANCE_SCORE でフィルタし上位 MAX_SCRIPT_ARTICLES 件）、使用済みマーク
episode_service.pyエピソードの CRUD・状態/フェーズ更新・期限切れ取得・generating スロットの確保（部分一意Index を利用した claim_generating_slot）。SQLite ロック競合に対する指数バックオフ付きリトライを全書き込みに適用
ffmpeg_service.pyffmpeg ラッパー。WAV結合（concat）、WAV→MP3、任意音源→WAV正規化（24000Hz/mono/s16 等へリサンプル）、add_jingles_and_encode（filter_complex で opening / ending を atrim=end=jingle_duration + フェードイン/アウト処理して本編と concat → MP3）
replacement_table.pyTTS 用の読み置換テーブル。英略語や記号など読み誤りしやすい表記を発音用テキスト（spoken_text）へ変換。表示用（display_text）とは分離して script.json に保持

generate_script.py — 台本生成の内部構造

  1. Narrative Arc 生成 — 要約一覧を generate_narrative_arc.md プロンプトに渡し、エピソード全体の共通テーマ・テーマ概要・記事の紹介順序を先に決定。失敗時はアーク無しで続行
  2. 台本生成 — generate_radio_script.md にアーク情報と要約JSONを注入し、lines（speaker / text / article_id / section）を生成。section は intro / news / transition / discussion / outro の5種に正規化
  3. トランジション保証（_ensure_transitions） — article_id の切り替わり境界に transition 行が無ければ、約40種のフレーズプール（news用）と討論導入フレーズ（discussion用）から直前と重複しないよう乱択して挿入。話者は直近3行の偏り（同一話者2連続、男女バランス）を見て自動選択
  4. 自動Linter（lint_script） — 禁止フレーズ、テキスト完全重複、口癖フレーズの2回以上使用、「数字で見ると」と言いつつ数字が無い行、discussion が news より前に出現する構成違反を検出。エラーがあれば前回出力と指摘事項を添えた修正プロンプトで再生成
  5. 採用記事のマーク — 台本に採用した記事を status=used に更新し、翌日以降の重複採用を防止

review_script.py — 4ディレクターレビュー

台本JSONを4種類のキャラクタープロンプト（review_genius_director.md ほか）に通して個別レビューを収集し、review_synthesize.md で全フィードバックを統合した改訂版台本を生成します。結果は dict（revised / review_count / revision_summary / lines_count）で返り、呼び出し側は例外を捕捉する必要がありません。改訂版は別の episodes レコードとして合成・ビルドされるため、同じ日付にオリジナル版とレビュー版の2エピソードが並ぶ設計です。

synthesize_voicevox.py — 音声合成の細部

  • WAVファイル名は無音・ジングル挿入分も含む通し番号（001.wav〜）。transition 行の直前にトランジションジングル（24000Hz/mono/s16 へ ffmpeg 正規化してコピー、失敗時は wave モジュールで無音1秒を生成）を挿入
  • 合成成功行には display_text（画面表示用）/ spoken_text（置換適用済み）/ wav_file を書き戻し、script.json を更新
  • TTSエンジン切替（VOICEVOX 24000Hz ↔ AivisSpeech 44100Hz）によるレート不一致を防ぐため、最後に lines/ 内の全WAVを最多数派レートへ一括リサンプル

build_episode.py — 組み立てとタイムコード

  • lines/ 配下のWAVをソートして結合し、add_jingles_and_encode でオープニング/エンディングジングル（既定10秒・フェード1秒）を付加した episode.mp3 を生成
  • 各セリフ行に start_time（秒）を付与。オープニングジングルの実長（または jingle_duration の小さい方）をオフセットとし、WAV長を累積して算出。wav_file 欠落行（旧エピソード）には file_counter ロジックを再現して推定
  • metadata.json（title / duration_seconds / audio_path / created_at）を出力し、API の一覧・詳細補完に利用

運用・エラー処理

ログ

  • バッチ: logging_config.setup_daily_logging により data/logs/YYYY-MM-DD.log へ集約。cron の標準出力は data/logs/crontab.log に追記され、entrypoint.sh が Docker ログへ tail する
  • 最終実行の結果は data/run_manifest.json（done / skipped_no_new / error_summarize: ... 等）に保存され、外形監視に利用できる

障害と復旧

事象挙動・対処
Ollama / TTS 停止run_daily は事前ヘルスチェックで ERROR をログし続行（後続ステップで失敗 → status=failed）。/health/ollama・/health/voicevox で個別確認可能
新着記事 0 件run_daily は skipped_no_new として exit 0（エピソードは作られない）。orchestrate は既存の要約済み記事があれば続行
LLM の不正 JSONOllamaClient が強制JSONプロンプトで自動リトライ。要約では該当記事を error にしてスキップ、台本では Linter 修正ループで再生成
generating の滞留API がコンテナ再起動等で中断した場合、次回 POST /generate 時に同日付の stale レコードをリセットして再利用。手動復旧用に scripts/reset_stale_generating.py も同梱
SQLite ロック競合EpisodeService の書き込みは指数バックオフ付きで最大5回リトライ
ディスク逼迫cleanup_episodes.py が保持期間（既定30日）超過のエピソードを DB・ファイルとも削除（ID ベース／旧日付ベース両ディレクトリに対応）

テスト

  • backend/tests/: 生成フローと stale generating リセットの pytest
  • app/batch/test_ensure_transitions.py / app/services/test_fetch_summaries_for_script.py: トランジション挿入ロジックと台本用要約取得のユニットテスト
