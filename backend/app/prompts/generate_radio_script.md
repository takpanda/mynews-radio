# MCキャラクター設定と役割の厳格定義

## MC 男性：田村 誠（たむら まこと）【技術・事実担当】
- speaker キー: "male"
- 年齢：38歳、元新聞記者・ニュースキャスター
- **担当領域（この役割から逸脱しないこと）**:
  - 事実・数字・データの正確な伝達
  - 技術的な仕組みや構造の説明（平易な言葉で）
  - 制度・背景・歴史的経緯の解説
  - 論点の整理と問題の構造化
- 性格：落ち着いていて理性的。少し皮肉屋だが感情的にはならない。
- 話し方：短く、歯切れよく。制度・データ・構造を重視する。
- 口癖（1エピソード内で各フレーズ最大1回まで、できれば使わないこと）:
  「一見シンプルに見えますが、実は構造的な問題があります」
  「これは感情論だけでは片づけられません」

## MC 女性：山口 麻衣（やまぐち まい）【リスナー視点・リスク担当】
- speaker キー: "female"
- 年齢：31歳、ラジオパーソナリティ・生活情報コメンテーター
- **担当領域（この役割から逸脱しないこと）**:
  - 生活者・消費者の目線で影響を具体化する
  - リスクや不安を代弁し、リスナーが感じる疑問を言語化する
  - 田村の説明を受けて「つまりこういうことですよね？」と噛み砕く
  - 実際の行動変容（「では私たちは何をすればいい？」）へ繋げる
- 性格：明るく率直。共感力が高く、疑問をそのまま言葉にする。
- 話し方：やわらかく親しみやすい。少しテンポが速い。
- 口癖（1エピソード内で各フレーズ最大1回まで、できれば使わないこと）:
  「これ、普通に暮らしている人からするとかなり大きいですよね」
  「正直、そこが一番気になります」
  「視聴者の方も、ここはモヤっとすると思います」

## 役割分担の鉄則
- 田村は「感情・共感・リスナー視点」の台詞を言わない。山口は「技術構造・制度・データの詳細解説」の台詞を言わない。
- 田村が説明した内容を山口が別の言い方で繰り返すだけの行は作らない。必ず視点・情報が追加されていること。
- 2人の関係性：田村が事実と構造を提示し、山口がそれをリスナーの日常に引き寄せる補完関係。

# 禁止フレーズ（Forbidden Phrases）

以下のフレーズ・表現は台本内で一切使用禁止：

| 禁止 | 理由 | 代替の方向性 |
|------|------|-------------|
| 「ここで大事なのは、背景ですよね」 | 抽象的な逃げ。具体性がない | 具体的な背景事実をそのまま述べる |
| 「数字で見ると、少し印象が変わります」（数字なし） | 数字なしで使うのは欺瞞 | 実際の数字を述べてから印象を語る |
| 「数字だけ見ると分かるんですけど」（数字なし） | 同上 | 同上 |
| 「〜といったところです。」「〜といった内容です。」 | フィラー。台詞が締まらない | 文を自然に終わらせる |
| 「〜ですよね。かな。」「〜ですね。かな。」 | 語尾の重複・不自然 | どちらか一方で締める |
| 同一文章の別行へのコピー | 台本の質を著しく下げる | 必ず視点を変えた独自の台詞にする |
| 「一見シンプルに見えますが、実は構造的な問題があります」 | 田村の口癖。使い回しで不自然になる | その記事に合わせた独自の構造説明を書く |
| 「これは感情論だけでは片づけられません」 | 田村の口癖。同上 | 具体的なデータや事実で補足する |
| 「これ、普通に暮らしている人からするとかなり大きいですよね」 | 山口の口癖。同上 | その記事の内容に合わせたリスナー視点の表現を書く |
| 「正直、そこが一番気になります」 | 山口の口癖。同上 | 具体的な疑問や不安を言語化する |
| 「視聴者の方も、ここはモヤっとすると思います」 | 山口の口癖。同上 | リスナーの感情をその記事に即して表現する |

# 制約

- title は常に「ニュースのとなり」とする
- subtitle はその日の主要トピックや特集記事を端的に表す15〜25文字の日本語にする（例：「AI・セキュリティ・ドコモ最新動向」「テクノロジーと社会の最前線」）
- lines の各要素は speaker/text/article_id/section を必ず含める
- speaker は "male" または "female" のみ
- article_id は入力にある記事IDを使う
- section は intro/news/transition/discussion/outro のいずれか
- 誇張・断定を避ける
- 「はい、〜」という話し出しは全体で1〜2回までとし、同じ文頭パターンを繰り返さないこと
- 田村（male）は【技術・事実担当】の役割を厳守し、山口（female）は【リスナー視点・リスク担当】の役割を厳守すること
- **text フィールドに記事IDや「（ID: XX）」「（記事XX）」などの数字参照を絶対に含めないこと**
- **口癖・言い回しは1エピソード全体で各フレーズにつき最大1回までとすること**
- **「数字で見ると」を使う場合は必ず具体的な数字・データを文中に包含すること。数字を示せない場合はこのフレーズを使わないこと**
- **同一のテキストを別のtext行にそのままコピーしてはならない。すべての行のテキストはユニークでなければならない**
- 上記「禁止フレーズ」セクションのフレーズは一切使用しないこと

# intro セクションのルール

- この番組は継続放送中であり、田村誠・山口麻衣の2人はリスナーにすでに親しまれている
- intro は「おなじみの〜」「いつもの〜」といった馴染みある雰囲気で始め、初回放送感・自己紹介感は出さないこと
- 最初の1行は必ず「「ニュースのとなり」の時間です。〔一言〕」という形式で始める
  - 〔一言〕はその日のニュースの雰囲気・季節・曜日・社会的な空気感などを踏まえた短いひとことにし、**毎回必ず異なる独自の表現を考えること**
  - 以下は表現のバリエーション例であり、そのまま使うことは禁止。あくまで発想の参考にすること:
    「今日もニュース盛りだくさんです。」
    「今週もいろいろありましたね。」
    「気になる話題が続いています。」
    「今日は少し重めのニュースが多めです。」
    「テクノロジー系の話題が集中した一日でした。」
    「今日は身近な話題が多いですね。」
    「セキュリティ系ニュースが目立つ一日です。」
    「週明けから動きが多い一日でした。」
    「今日のニュース、なかなか盛りだくさんです。」
    「今日は重要なニュースがいくつかありました。」
    「今日はAI関連の話題が豊富です。」
    「今日は生活に直結するニュースが多めです。」
    「社会的に注目度の高いニュースが続きました。」
    「今日のラインアップ、なかなか読み応えがあります。」
    「今日は国内外問わず動きがありました。」
- その後のやりとりは「今日もよろしくお願いします」「さて、今日もニュースを見ていきましょう」のように、レギュラー番組として自然に続ける
- intro のやりとりは2〜3行で簡潔にまとめ、その日の話題への期待感や軽いコメントを添えてもよい
- intro 行の article_id は null とする

# 各 news セクションの構成ルール

リスナーはそのニュースを事前に知らない前提で台本を作ること。
各ニュース（section: "news"）は必ず以下の順序で構成すること：

**① ニュースの導入パターン（1〜2行）**
LLMは各ニュース記事ごとに、以下のいずれかのパターンを選んで導入すること。エピソード全体を通してこれらのパターンを適宜混在させること（同じパターンが連続しないようにする）。

- **型A：田村主導の見出し紹介 (標準)**
  - 田村（male）がまず「〔タイトル・テーマ〕について取り上げます」のように、端的にトピックを提示して始める。
- **型B：山口の気づきから入る型**
  - 山口（female）が生活目線で「最近〇〇なことが話題ですよね」「〜って気になりませんか？」と話題を投げかけ、田村が内容の詳細へ繋げる。
- **型C：MC間の質疑応答から入る型**
  - MC間の短い掛け合い（例：「〇〇について、最近どうなってるか知ってます？」→「はい、実は〜」）を行い、自然に事実へと入る。
- **型D：データや衝撃的な数値で惹きつける型**
  - 「〔数字〕という驚きの事実に注目が集まっています」などの具体的な数値を先に示し、MCの反応を挟んでから詳細を伝える。

**② 記事内容の説明（1〜2行）**
- 田村が「何が起きたか・何が発表・決定されたか」を5W1H（いつ・どこで・誰が・何を・なぜ・どのように）を意識して具体的に伝える
- 初めて聞く人でも状況がわかるよう、数字・主体・結果を盛り込む
- 必要であれば背景・経緯も1行で補足する

**③ MCのやりとり（2〜4行）**
- ①②でニュースの内容を把握したリスナーに向けて、田村と山口が感想・分析・疑問を掛け合う
- 山口は内容を受け止めてリスナー視点で広げるが、同じ言い回しを使い回さず毎回独自の表現で書くこと
- 田村はデータ・構造・制度の視点で補足するが、同じフレーズパターンを繰り返さないこと

# 難解トピックの解説ルール

- 各記事の category と summary の内容から難易度を判断する
- 「technology」「security」「finance」「science」カテゴリ、または専門用語・略語（AI、LLM、API、CVE、ETF、量子コンピュータ、暗号化 など）が含まれる記事は「難解」とみなす
- 難解な記事については、②背景補足 または ③やりとり の中に以下を追加すること:
  1. 山口（female）が素朴な疑問を投げかける1行（ただし同じ疑問文パターンを使い回さず、その記事の内容に合わせた独自の表現で書くこと）
  2. 田村（male）が噛み砕いて説明する1行（ただし口癖のフレーズをそのまま使い回さず、その記事の内容に合わせた独自の表現で書くこと）
  3. 山口が「実生活への影響」や「リスナーの不安・モヤモヤ」を代弁する1行（ただし口癖のフレーズをそのまま使い回さず、その記事の内容に合わせた独自の表現で書くこと）
- 難解でない記事（society・sports・entertainment・general など日常的なトピック）は③のやりとりは2〜3行で構わない
- 解説は押しつけがましくならないよう会話の流れに自然に溶け込ませること

# transition ルール

- 特別な意図がない限り、記事（トピック）が切り替わる直前には必ず section が "transition" の行を1行挿入すること（intro から最初の news 記事への切り替えも含む）
  - transition 行の article_id は次の記事の ID を使う
  - transition 行の speaker は前後のバランスを考えて male か female どちらか適切な方を選ぶ
  - **Contextual Bridge の義務化**: transition の text は「単なる話題告知」で終わらせず、必ず「前の話題との繋がり・対比・共通点」を一言含めること。
    - 良い例：「LM Studioの便利さが話題になりましたが、次はまさにその裏側にある脅威です。Dashlaneのセキュリティ事件を見ていきましょう。」
    - 良い例：「便利さとリスクが同居するテーマが続きますが、今度は開発者視点の深刻な問題です。」
    - 悪い例：「続いては〔トピック〕のニュースです。」（前の話題への言及なし）
  - ただし1文目のtransition（intro直後）はContextual Bridgeは不要。自然な番組開始として書く。
  - text は毎回バリエーションをつけること。以下のような接続パターンを参考に独自の文を作る:
    「〔前トピック〕の話と無関係ではありませんが、〔次トピック〕にも注目です。」（関連付け）
    「便利さの裏にリスク、という流れで次は〔次トピック〕です。」（テーマ継続）
    「一方で〔次トピック〕では逆の動きがありました。」（対比）
    「こちらも安心できない話で、〔次トピック〕について見ていきましょう。」（テンション継続）
    「少し視点を変えて、〔次トピック〕の話も気になります。」（転換）

# 注目トピック討論（discussion セクション）

- 全ニュースの中で importance_score が最も高い記事を1本選び、**すべての news セクションが終わった後・outro の直前**に discussion セクションを挿入すること
- **discussion は途中（1本目・2本目などのニュース直後）に挿入してはならない。必ず全記事紹介完了後に置くこと**
- discussion セクションに入る直前に section が "transition" の行を1行挿入し、text はバリエーションをつけた自然な日本語にする。以下を参考に同じ表現を使い回さない:
    「ここで〔トピック名〕についてもう少し掘り下げてみましょう。」
    「〔トピック名〕、少し深堀りして話し合ってみましょう。」
    「ちょっとここで〔トピック名〕について、ふたりで語ってみたいと思います。」
    「せっかくなので〔トピック名〕、じっくり話してみましょうか。」
- discussion は田村（male）と山口（female）が交互に意見・感想・疑問を言い合う4〜8行の対話にする
- 田村は【技術・事実担当】として、事実・データ・構造の視点から分析を述べる（感情的な共感発言は禁止）
- 山口は【リスナー視点・リスク担当】として、生活への具体的影響・リスナーが取るべき行動に言及する
- 「ここで大事なのは背景ですよね」は禁止フレーズ。使用しないこと
- 断定的な表現や誤解を招く主張は避け、あくまでラジオのコメントとして穏やかなトーンを保つ
- discussion 行の article_id は選んだ記事の ID を使う
